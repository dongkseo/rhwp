## 1. 문제 요약

PR #1420에서 rhwp-studio의 `system | light | dark` 테마 기반과 주요 UI chrome 다크테마가
도입되었지만, 개별 다이얼로그와 팝업에는 inline style 기반 라이트 테마 색상과 공통 폼 컨트롤의
토큰 누락이 남아 있다.

현재 관찰된 문제는 다음으로 묶인다.

1. 표/셀 속성의 읽기 전용 크기 입력 필드가 흰색 계열 배경으로 남아 다크테마와 어긋난다.
2. 표/셀 속성 및 셀 테두리/배경의 선 샘플, 미리보기 주변 버튼, 보조 선 색상에 라이트 테마
   하드코딩이 남아 있다.
3. 수식 편집 미리보기는 다크 UI 표면 위에 실제 검은 수식 색상을 그대로 그려 가독성이 낮다.
4. 쪽 테두리/배경은 중앙 문서 종이 미리보기는 유지해야 하지만, fieldset/legend와 SVG 사방 버튼은
   다크모드 대비가 부족하다.
5. 표 만들기 팝업, 문단 모양, 미주 모양, 검증 모달, grid 설정 등 일부 UI에 라이트 테마 색상이 남아 있다.
6. Chrome `Auto Dark Mode for Web Contents`가 켜진 환경에서 `보기 > 테마 > 밝게`를 선택해도
   브라우저 강제 다크 변환이 적용되어 앱의 명시적 밝은 테마 의도와 표시가 불일치한다.

## 2. 목표

1. #1420에서 만든 테마 체계를 유지하면서, 잔여 다이얼로그/팝업 UI를 semantic token 기반으로 정리한다.
2. 다크모드 적용 대상인 UI chrome과 실제 문서 색상/미리보기 영역을 명확히 분리한다.
3. 표/셀 속성, 셀 테두리/배경, 수식 편집, 쪽 테두리/배경을 우선 수정 대상으로 삼는다.
4. 동일 패턴의 라이트 테마 하드코딩을 함께 점검해 후속 재발 가능성을 낮춘다.
5. Chrome Auto Dark Mode 환경에서도 rhwp-studio의 명시적 테마 선택이 가능한 범위에서 우선되도록 한다.
6. 기존 light 테마와 실제 문서 색상 의미를 보존한다.

## 3. 다크모드 처리 원칙

### 3.1 다크모드 적용 대상

- 다이얼로그 배경, 탭, fieldset/legend, 일반 라벨, 버튼, select, input, textarea
- 비활성/읽기 전용 필드의 배경, 글자색, 테두리
- 미리보기 주변 조작 버튼, 방향 버튼, 보조 라벨, 일반 UI 테두리
- 표 만들기 quick grid popup처럼 문서 색상 의미가 없는 일반 UI 팝업

### 3.2 문서 색상 유지 대상

- 편집 종이와 실제 문서 배경
- 쪽 테두리/배경 중앙 SVG 문서 미리보기의 흰 종이 표현
- 셀 배경 없음 또는 문서 종이 배경을 표현하는 흰색 preview 영역
- 사용자가 지정한 실제 선 색, 채움색, 색상 견본
- 인쇄, 내보내기, 문서 렌더링 결과에 영향을 주는 색상

### 3.3 수식 미리보기 원칙

수식 색상 자체를 다크모드에서 흰색으로 저장하거나 preview-only로 임의 반전하지 않는다. 기본 방향은
수식 미리보기 배경을 문서 종이 배경에 맞춰 기본 검은 수식이 보이게 하는 것이다. 이 방식은 저장되는
수식 색상 의미와 화면 가독성을 함께 보존한다.

### 3.4 Chrome Auto Dark Mode 대응 원칙

`밝게`, `어둡게`, `시스템`이라는 앱의 명시적 테마 선택이 브라우저 자동 색상 변환보다 우선되어야 한다.
특히 `밝게` 선택 시에는 앱이 라이트 UI를 의도했다는 힌트를 브라우저에 명확히 제공한다. 단, Chrome
실험 기능 자체의 전역 동작이나 사용자 브라우저 설정을 rhwp-studio가 임의로 변경하지는 않는다.

## 7. 수행 단계 후보

구현계획서에서는 다음 후보를 3~6개 단계로 정리한다.

1. 공통 폼/다이얼로그 토큰 기반 정리
   - `.dialog-input`, `.dialog-select`, textarea, disabled/read-only/focus 상태를 우선 정리한다.
   - 개별 다이얼로그의 inline style과 충돌하지 않는지 확인한다.
2. 주요 재현 다이얼로그 수정
   - 표/셀 속성, 셀 테두리/배경, 수식 편집, 쪽 테두리/배경을 우선 처리한다.
   - 문서 종이 preview와 실제 색상 견본은 별도 토큰 또는 절대색으로 의도 보존한다.
3. 추가 후보 sweep
   - 표 만들기 popup, 미주 모양, 문단 모양, toolbar popup, validation/grid dialog를 점검한다.
   - 문서 preview 성격인 값과 UI chrome 성격인 값을 분류한 뒤 필요한 최소 범위만 수정한다.
4. 회귀 가드 보강
   - 기존 `rhwp-studio/e2e/theme-mode.test.mjs`를 유지하고, 필요한 경우 focused e2e/DOM 검증을 추가한다.
   - 최소한 dark mode에서 주요 dialog 내부 색상과 문서 종이 preview 유지 여부를 검증한다.
5. Chrome Auto Dark Mode 최종 점검
   - 앞선 UI 토큰 정리 후 마지막 단계에서 Chrome Auto Dark Mode 환경을 검증한다.
   - 필요한 경우 `color-scheme`, meta theme-color, root dataset 적용 순서를 보정한다.

## 8. 검증 계획

필수 검증 후보:

```bash
cd rhwp-studio && npm run build
cd rhwp-studio && node e2e/theme-mode.test.mjs --mode=headless
```

필요 시 추가:

```bash
cd rhwp-studio && node e2e/<신규-다크모드-dialog-검증>.test.mjs --mode=headless
```

시각 확인:

- 다크모드에서 표/셀 속성, 셀 테두리/배경, 수식 편집, 쪽 테두리/배경의 대비 확인
- 쪽 테두리/배경 중앙 문서 preview가 흰 종이 표현을 유지하는지 확인
- 수식 편집 preview에서 기본 검은 수식이 읽히는지 확인
- 표 만들기 popup, 미주 모양, 문단 모양 등 추가 후보의 라이트 하드코딩 잔여 확인
- Chrome Auto Dark Mode 활성화 환경에서 `보기 > 테마 > 밝게` 선택 시 앱의 밝은 테마 의도가 유지되는지 확인

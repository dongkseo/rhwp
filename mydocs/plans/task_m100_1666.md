# Task M100 #1666 수행 계획서

## 개요

- 이슈: #1666 `[CI] PR용 Rust profile 재검토: --release vs release-test`
- 부모 이슈: #1668 `[Tracking/RFC] CI Build & Test 실행 시간 단계별 단축`
- 마일스톤: M100 / v1.0.0 조판 엔진 체계화
- 작업 브랜치: `task1666-ci-release-test-profile`
- 기준 결정: #1668 메인테이너 코멘트 `2026-06-30T07:11:16Z`

## 목적

PR CI의 `Build & Test` wall time을 줄이기 위해 PR 검증을 `release-test` profile 중심으로 전환한다.
현재 `release` profile은 `lto = true`, `codegen-units = 1`이라 PR 피드백 루프에는 무겁다. 이미 정의된
`release-test` profile은 `release`를 상속하되 `lto = false`, `codegen-units = 16`이라 테스트 목적에 더 적합하다.

## 핵심 가설

#1666은 단순히 명령 문자열을 `--release`에서 `--profile release-test`로 바꾸는 작업이 아니다. 다음 두 비용이
실제로 줄어드는지 확인하는 작업이다.

1. profile 불일치 비용
   - 현재 PR `Build`, `Native Skia tests`, `Run lib tests`는 `release` profile이고, `Run integration tests`는
     `release-test` profile이다.
   - 이 조합에서는 같은 crate라도 step 간 profile이 달라 별도 산출물과 재컴파일이 발생할 수 있다.
   - #1666에서는 PR의 build/lib/integration test 축을 `release-test`로 맞춰 profile 차이로 인한 재컴파일이
     줄어드는지 확인한다.

2. release LTO/link/codegen 비용
   - `release` profile은 `lto = true`, `codegen-units = 1`이라 link/codegen 비용이 크다.
   - `release-test`는 `lto = false`, `codegen-units = 16`이므로 PR 검증에서 link/codegen 비용을 낮출 수 있다.
   - after 보고에서는 단순 wall time뿐 아니라 cargo 로그의 `Finished release` / `Finished release-test`와
     step별 compile 양상을 함께 비교한다.

기대값:

- `Build`, `Run lib tests`, `Run integration tests`가 같은 `release-test` profile 산출물을 더 많이 공유한다.
- `Finished release ...`가 PR의 일반 build/test step에서 사라지고 `Finished release-test ...` 중심으로 바뀐다.
- `Compiling rhwp` 또는 `Dirty rhwp ... has changed`가 발생하는 step 수나 위치가 줄어드는지 확인한다.
- `Native Skia tests`는 `native-skia skia` feature set이 다르므로 별도 compile이 남을 수 있다. 이 경우는
  실패가 아니라 feature 차이로 인한 정상 비용으로 분리해 기록한다.

## 메인테이너 결정사항 반영

### PR release-grade 검증 정책

- 결정: PR은 `release-test` 중심으로 빠른 회귀 검출을 수행한다.
- 결정: release-grade 검증은 `devel` push / tag 또는 release workflow로 이동한다.
- #1666 적용: PR의 `Build`, `Native Skia tests`, `Run lib tests`를 `release-test` profile로 전환한다.
- #1666 적용: `Run integration tests`는 이미 `release-test` profile이므로 명령 의미를 유지한다.

### 회귀 가드 1:1 추적성 보존

다음은 하지 않는다.

- 통합 테스트 파일 통합
- 회귀 가드 명명 규칙 변경
- `tests/golden_svg/issue-NNN/` 자산 구조 변경

`tests/*.rs` 162개와 issue 계열 131개 회귀 가드가 PR마다 계속 실행되는지 after 보고에 포함한다.

실제 변경 PR 구성:

- 코드 PR에 포함:
  - `.github/workflows/ci.yml`
  - #1666 수행계획서 / 구현계획서 / 단계 보고서 등 절차 문서
- 코드 PR에 포함하지 않음:
  - 확정 전 measurement 원천 문서
  - 부모 #1668 장기 추적 문서의 after 수치 갱신

  - PR CI와 merge 후 `devel` run 관측값
  - before/after raw table

코드 PR 본문에는 "measurement 원천 문서는 CI 관측 후 후속 문서 PR에서 갱신"한다고 명시한다.

`Cargo.toml` profile 정의 변경이 필요해지는 경우에는 메인테이너 요청대로 단독 PR로 분리한다. 현재 계획상
`release-test` profile은 이미 정의되어 있으므로 `Cargo.toml` 변경은 하지 않는다.

## 현행 상태

### Cargo profile

`Cargo.toml`에는 이미 다음 profile이 정의되어 있다.

| profile | 주요 설정 | 해석 |
|---------|-----------|------|
| `release` | `lto = true`, `codegen-units = 1`, `strip = "debuginfo"` | release-grade 검증. PR에는 무거움 |
| `release-test` | `inherits = "release"`, `lto = false`, `codegen-units = 16` | 테스트용 release 변형. #1666 주 적용 대상 |

따라서 1차 구현에서는 `Cargo.toml` 변경 없이 `.github/workflows/ci.yml` 명령 전환만 검토한다.

### 현재 `CI / Build & Test` 명령

| step | 현재 명령 | #1666 판단 |
|------|-----------|------------|
| Build | `cargo build --release --verbose` | PR에서는 `--profile release-test` 후보 |
| Native Skia tests | `cargo test --release --features native-skia skia --lib --verbose` | PR에서는 `--profile release-test` 후보 |
| Run lib tests | `cargo test --release --lib --verbose` | PR에서는 `--profile release-test` 후보 |
| Run integration tests | `cargo test --profile release-test --tests --verbose` | 이미 적용됨. 유지 |
| Clippy | `cargo clippy -- -D warnings` | 이번 범위 밖. 유지 |

## 구현 방향

구현 계획서 승인 후 다음 방향으로 검토한다.

1. PR의 일반 build/test step을 `release-test` profile로 전환한다.
   - `cargo build --profile release-test --verbose`
   - `cargo test --profile release-test --features native-skia skia --lib --verbose`
   - `cargo test --profile release-test --lib --verbose`
   - `cargo test --profile release-test --tests --verbose` 유지
2. release-grade 검증을 trusted event로 보완한다.
   - `devel` / `main` push에서 `cargo test --release --features native-skia skia --lib` 유지 또는 별도 step 추가
   - `devel` / `main` push에서 `cargo test --release --lib`와 `cargo test --release --tests` 유지 또는 별도 step 추가
   - tag / release workflow의 `cargo build --release` 검증 경로를 확인한다.
3. `Build & Test` job 이름과 required check 표면은 유지한다.
4. job 병렬화, `Swatinem/rust-cache`, nextest 도입은 하지 않는다. 각각 #1665, #1667 또는 후속 이슈 범위다.

### cargo 로그 측정 기준

wall time만으로는 #1666의 효과를 충분히 설명할 수 없으므로, after 보고에는 step별 cargo 로그 해석을 함께
포함한다.

| 관측 항목 | 확인 목적 |
|-----------|-----------|
| `Compiling rhwp` | local crate compile이 어느 step에서 발생하는지 확인 |
| `Fresh ...` | cache/fingerprint hit로 재사용된 crate가 늘었는지 확인 |
| `Dirty rhwp ... has changed` | step 간 profile/feature 차이로 dirty 판정이 남는지 확인 |
| `Finished release ...` | PR 일반 build/test에서 release profile 사용이 남았는지 확인 |
| `Finished release-test ...` | build/lib/integration test가 release-test로 정렬됐는지 확인 |

step별 기대 해석:

| step | after 기대값 | 남을 수 있는 비용 |
|------|--------------|-------------------|
| Build | `Finished release-test ...`로 전환 | 최초 local crate compile은 남을 수 있음 |
| Native Skia tests | `Finished release-test ...`로 전환 | `native-skia skia` feature set 차이로 별도 compile 가능 |
| Run lib tests | `Finished release-test ...`로 전환 | Build 산출물과 공유되면 compile 감소 기대 |
| Run integration tests | 기존처럼 `release-test` 유지 | test binary link 비용은 162개 가드 실행 때문에 남음 |

before/after 보고에는 다음을 함께 기록한다.

- step별 `Compiling rhwp` 발생 여부
- step별 `Finished release` 또는 `Finished release-test` 문자열
- `Dirty rhwp ... has changed`가 발생한 파일과 step
- feature 차이로 설명되는 compile과 profile 불일치로 설명되는 compile의 분리
- wall time 감소가 compile 감소인지 link/codegen 비용 감소인지에 대한 해석

- `tests/`와 `tests/golden_svg/` 변경 없음 확인

### PR CI 관측

- `Build & Test` 성공 여부
- `Build`, `Native Skia tests`, `Run lib tests`, `Run integration tests` 모두 실행 여부
- `Build`, `Native Skia tests`, `Run lib tests`가 `release-test` profile로 실행되는지 cargo 로그 확인
- `Compiling rhwp`, `Fresh`, `Dirty rhwp`, `Finished release`, `Finished release-test` 로그 비교

- read-only / reservation / save failure 경고 여부

### trusted branch 관측

코드 PR merge 후 `devel` push run에서 다음을 확인한다.

- PR에서 빠진 release-grade 검증이 trusted event에서 실행되는지
- release-grade step 실패 시 원인 위치가 충분히 드러나는지
- cache save 정책이 #1664 이후 상태를 유지하는지

## 리스크와 대응

| 리스크 | 대응 |
|--------|------|
| PR green이 release green을 더 이상 직접 의미하지 않음 | `devel` push / tag / release workflow에 release-grade 검증 유지 |
| LTO/link-only 회귀를 PR에서 놓칠 수 있음 | trusted branch와 tag에서 `--release` 검증을 수행하고 문서에 의미 차이를 명시 |

| cache 상태가 측정값을 흔들 수 있음 | #1664 cache 기준선과 함께 hit/miss/save 상태를 항상 기록 |
